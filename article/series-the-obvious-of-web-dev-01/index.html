はじめに - あるいはここでの「あたりまえ」について -

常識とは、18歳までに身に付けた偏見のコレクションである。
— Albert Einstein1

仕事中に「こんなこと、分かっていて当たり前だろ？」「当然できますよね？」と誰かが言ったのを聞いたことがあるだろうか。

企業及び業務における、人間の能力に対する「あたりまえ」というのは、ある程度保証されていると考えられている。
企業面接や紹介会社などのクオリティゲートを通り採用された人間は、当然のように周囲から「あたりまえ」のことができるものとして期待されるし、企業固有の「あたりまえ」は入社初期に研修や集中的な学習プランによって叩き込まれ、そこから現場に投入されるものと考えられているからだ。しかし、実際には「あたりまえ」のことが出来ていないがゆえに、誰かに対する失望の籠った声が発せられることが少なくない。そして、「あたりまえ」を知らない人間は、業務参加の資格や機会を暗黙的に失っていき、こっそりと仕事ができない人のラベルを貼られることになる。

これは何が悪いのだろうか？必要なスキルセットを選定することができなかった人事部門？ずさんな研修を行った研修担当者？それとも、不良在庫のような人材を押し付けた紹介会社？あるいは「あたりまえ」を知る努力さえ放棄した仕事のできない人？2
……広い世の中を探せばそういった個人の能力不足や怠慢が原因のケースもあるだろうが、僕としては別要因をひとつ提示したい。
「誰かのあたりまえが、みんなのあたりまえじゃなかった」
ただこの一点である。

どういうことかを説明するために、まず「あたりまえのあたりまえ」について話しておきたい。業務における「あたりまえ」とは何なのだろうか？
各々自分の認識を持っているだろうが、このシリーズでは各自の認識を抑えて、以下の定義に従ってもらいたい。

  • 「あたりまえ」とは、ある集団における特定業務遂行に必須となる最低限の知識の総称である。
  • 「あたりまえ」の知識には、コミュニケーション上必要な語彙・プロセス理解・背景理解が含まれる。

そして、この定義として捉えた時に顕出する「あたりまえ」の原則を2つ紹介する。

  1. 「あたりまえ」は変わるものである。すべてがまるごと変わることもあれば、一部だけが変わることもある。

50年ほど前のエンジニアの業務の「あたりまえ」を考えてみよう。
複数の巨大な真空管で構成された電算機に計算をさせるために、命令をパンチカードに打ち込み、電算室を利用する許可を得て読み込ませ、プログラムの実行を行う。文字一つを出力するためにも、多大なコストがかかるというのがかつての「あたりまえ」だった。

今はどうだろう。個人がコンソールを起動させ、echo "hogehoge"と打ち込む、あるいはブラウザの開発者ツールを開き、JavaScriptコンソールでconsole.log("hogehoge");と打ち込むだけで済む。誰の許可も必要ないし、実行ごとに莫大なコンピューティングリソースを投入することもない。
「プログラムを出力するためのコストは多大であるため、簡単なプログラムであっても、実行のたびにいちいち大仰な承認プロセスを経ることで不要な実行を抑制する」という、情報社会のあたりまえが大きく変わったのだ。

上記が労働体系や前提知識が丸ごと変わる大きな「あたりまえ」の変化だとすると、小さな変化はどうだろうか。
ここでは例として、JavaScriptにおける Babel.js と呼ばれるライブラリを例に挙げる。

JavaScriptには ECMAScript と呼ばれる世界共通の言語仕様書のようなものがあり、その仕様に従って各種JavaScriptの実行エンジンが作られ、ブラウザや他の実行環境に組み込まれることになる。
2015年、このECMAScript、およびJavaScriptに転換期が訪れた。ECMAScript 2015 (ES2015, ES6とも呼ばれる)のリリースである。このエディションの ECMAScript は多くの新しい構文や機能が追加されるもので、Webシステムの生産性に大きく貢献すると期待された。

……が、そうそううまくいかないのが現実だ。 ECMAScript はあくまで仕様であり、その仕様に準拠するかどうかは実行エンジンやブラウザの開発団体に大きく委ねられていた。その結果どうなったかはInternet Explorerの悪名を調べてみればよくわかるだろう。

Babel はこの混沌としたJavaScriptの仕様実装状況に、 「トランスパイル」 という手法で終止符を打ったライブラリである（また、トランスパイルを行うようなライブラリをまとめてトランスパイラと呼ぶ）。
これはあるプログラミング言語を別のプログラミング言語に書き直すという手法のことだ。
Babel の トランスパイル は、新しい仕様に準拠して書かれたJavaScriptのコードをまったく同じ動作をする古い構文に書き直す、というもので、クライアントサイドにブラウザ差異を吸収するようなライブラリ3を置くよりもずっとユーザー負担が少なく、しかも開発者にとってはよりわかりやすく使いやすい、新しい仕様に従って開発ができるというものだ。しかし一方で、これを実装に使うには、Webシステムの実装者にとって トランスパイル が「あたりまえ」のプロセスでないといけない。

時を経た2023年現在……トランスパイラは Babel 以外にも出現し、 ECMAScript の新旧変換だけでなく、他多くの変換処理を担うようになってきている。 トランスパイル を行うライブラリは労働体系や労働者のポジションを大きく変えるものではないが、しかし開発者が持つべき「あたりまえ」を確実に変えている。この特定領域の業務プロセスにおける変化が小さな「あたりまえ」の変化の例だ4。

  1. 「あたりまえ」は人と領域によって違う。

「あたりまえ」の定義に「ある集団」と書いてあることに気づいただろうか。
「特定業務遂行に必須となる最低限の知識」も重要だが、同様に重要なのがこの「ある集団」という記述だ。

この定義における「ある集団」とは、 特定の同一属性を持つ人間の集合のこと を指す。これを認識するために、一つの極端な例を出してみよう。
以下の文字列は、ある特定の集団にのみ理解できる文字列だ。

5LP -> 5LP -> 5LK -> (c) -> 236MK

あるいはこの数値の並びもそうだ。

108 - 130 - 95 - 80 - 85 - 102

最初に言っておくが、この文字や数字はまったくもってエンジニアリングや業務と関係ないので、詳しく意味を調べる必要はない5。僕と同じ、「ある程度ゲームをして、そのゲームの情報をインターネットで集めているゲーマー」集団に属している人にはわかるかもしれない、という、ある集団にとっては当然に読めるものだが、他にとっては何が書いてあるか意味がわからないものをチョイスしている。
この文字や数字を見た時、両方わかる・両方ともわからない、という人もいれば、上だけ、もしくは下だけ意味がわかる、という人もいるだろう。というのも当然で、上と下はまったく別のゲームの話である。「ある程度ゲームをして、そのゲームの情報をインターネットで集めているゲーマー」であっても、「ただし、ゲームAに関してだけであり、ゲームBに関してはまったくもって興味を持っていない」という組み合わせの属性を持つ人間はいくらでもいる。

この例を利用して「あたりまえ」について言いたいことは、 集団属性のくくり、つまり、ある対象の現実に対する各個人のスタンスや、対象への解像度や詳細度が異なれば異なるほど、「あたりまえ」は一致しない。 ということだ。ある集団が、どんな属性を持つか、何かの属性同士が重なった時に出てくる限定的な属性なのかなどを細やかに定義して周知しない限り、「あたりまえ」とは絶対に存在しえないものなのだ。

これらの原則から、 企業のような、属性・バックボーンが完全一致していない集団内において、それぞれに共通する「あたりまえ」は最初から少なく、人の数と時間に比例してさらに少なくなる。 ということが導き出される。ある集団のなかで「あたりまえ」の認識が最初から合っているかどうかはかなり怪しい。奇跡的に合っていたとしても、それぞれの人がある一つの「あたりまえ」を知っているか知っていないかというbitの組み合わせで表現してみるといい。
$$\frac{1}{2^{n} - 1}$$
として捉えられるだろう(誰も知らないことは、誰のあたりまえでもないから母集団から引く)。

例えば7人の人間がいて、それぞれの「あたりまえ」を持っているとき、全員が同じ「あたりまえ」を持っているケースは1/127しかない。7人中5人が知っているケースをあたりまえと算定してよいと考えても、全体のおおよそ1/5のケースしか存在しない。6
たった一つの「あたりまえ」でこれなのだから、複数の「あたりまえ」が一致するケースを考えると、もはや天文学的なケース数となる。奇跡的に全く同じ「あたりまえ」を持っていたとしてもこれだ。しかも「あたりまえ」はゆっくりと、しかし確実に時間とともに変化していくのだ。何もしなければ、今その集団が持っている「あたりまえ」が合うことなど絶対にない。

最初に提示した「仕事のできない人間」が作られる例に、これらの定義や原則を当てはめて考えてみよう。

まず、セクションごとに持つ「あたりまえ」は違うものになる。人事部門・研修担当者・紹介会社・現場というセクションで分けたとき、これら四つのセクションすべてで「現場において、業務遂行に必須となるコミュニケーション上必要な語彙・プロセス理解・背景理解」の認識が完全に合い、そして、これを完全に満たす人間を選定し、もし満たしていなければその能力を埋めるように研修を行うことなど可能なのだろうか？「フロントエンドエンジニアとして、あたりまえのことができる人が欲しい」という現場の声を聞き、採用・配置された人間が、特定プロダクトのトランスパイルを行う方法を含めた、その現場の「あたりまえ」をすべて知っていることを保証できるだろうか？
また、その「あたりまえ」について意思疎通を行ったのはいったいいつなのだろうか？例に挙げたフロントエンドに限らず、エンジニアリングの状況は小さな変化を繰り返して大きな変化を起こしており、5年と言わず、半年単位であたりまえがあたりまえでなくなってしまうことさえありうる。ジョブディスクリプションを確認してみたとき、必須スキルとして「jQuery 1.x」とは書いていないだろうか？7

はじめに提示した問題にあらためて答えよう。

しかし、実際には「あたりまえ」のことが出来ていないがゆえに、誰かに対する失望の籠った声が発せられることが少なくない。

これは何が悪いのだろうか？必要なスキルセットを選定することができなかった人事部門？ずさんな研修を行った研修担当者？それとも、不良在庫のような人材を押し付けた紹介会社？あるいは「あたりまえ」を知る努力さえ放棄した仕事のできない人？

「誰かのあたりまえが、みんなのあたりまえじゃなかった」 からだ。
そのうえ、 「みんなのあたりまえを作り、そして変えていくのは、人間にはそもそも難しい」 のだ。
何が悪いか？誰が悪いか？などはほとんど存在しない。どのような集団であっても、果てしないほどの注意を払って扱ったとしても、「あたりまえ」が言葉通りの意味を取ることは必ずと言っていいほどない。

これらを踏まえて、本シリーズにおける前提、つまり、本シリーズの「あたりまえ」を明示する。

何もせずに「あたりまえ」が「あたりまえ」であることはまずない。
「あたりまえ」を「あたりまえ」にするには、継続的な観測と、変化に追従する努力を必要とする。

……シンプルではあるが、完全に達成するのはとても難しい。それであっても、我々は「あたりまえ」を少しずつでも「あたりまえ」に近づけていく努力をしなければならない。
「あたりまえ」のことができないで、価値を生み出すことなどできないのだから。

複雑系と戦う武器：モデル

この文章は僕の持つ、Web開発技術の「あたりまえ」のメタモデルを紹介するものであるとも言える。

Webシステム開発、もっと大きなくくりで言うと、システム開発はよく 「複雑系(Complex System)」 を扱うものと言われる。複雑系とは……ちょっと一言で説明しづらい。ざっくりと扱うと、

  • いろんな細かな要素があって、連携しあっているものたちの総称
  • 系を構成する要素そのものも系になっている、いわゆるフラクタル構造をしている
  • 要素同士が相互に連携しあうことにより、単独要素では思いもよらなかった効果があらわれる特徴がある
  • 常に固定の要素で構成されるわけではなく、大小問わず要素そのものや要素の関係性に多くの変化が発生する

というような特徴を持つものだ8。
例えば、「経済」というものは、市場における金・人・モノ・サービスの動きの総体のことを指す。これを形成する一つ一つの要素はたとえば企業だ。企業は企業同士と連携してサプライチェーンを構築している。また、企業という要素も一つの系であり、企業内の部署同士の連携や、部署に所属する様々な人の活動によって構成される。そして、経済を内包する「現実世界」もまた、複雑系の一種となっている。
ある一つの企業や、所属する一人の人（要素）をそれぞれ観察し、その行動や特徴を並べても、経済（系）の動向を予測することはほぼ不可能なように、 複雑系は単一の要素の観測の積み重ねだけでは予測不可能なもの として扱われる。そのため、ある一定の前提のもとに現実の要素の一部を抜き取ったり、複雑系を構成する要素や要素同士の関係性の抽出・概念化などを行うことで、理論や定理などを構築しやすくしていく。この抽象化過程や試みのことを 「モデル化(Modeling)」 、モデル化された抽象的な概念のことを 「モデル」 と呼ぶ9。

このややこしい概念を利用して「あたりまえ」を形容すると、 特定集団において、ある系に対して各々が持っているモデルのうち、他人も同様に持っている要素や抽象、あるいはモデルそのもののことである。 と言える。人は独自モデル――この文では、 「メンタルモデル」 10としよう――を心の内面に持っており、自覚的・無自覚的どちらにかかわらずメンタルモデルを前提として話したり行動を行ったりする。つまり、人はそれぞれ独自のあたりまえを持っている。誰かのメンタルモデルが自然言語や図や文章といった形で明示化され、特定集団内に共通するメンタルモデルと一致するようになった状態になったときこそ、「独自のあたりまえ」が真に「あたりまえ」になったと言えるだろう。

普通、共通するメンタルモデルが1から作られることはほとんどない。誰かが考えたモデルを借りて少しずつ認識を固め、自分たちと合わない部分をカスタマイズをすることで、集団内の独自共通モデル = 「そこでのあたりまえ」を構築していくのがよくあるケースだ。
参考とすべきシステム開発における技術知識のモデルは多々あるものの、そのモデルを構成する要素や、前提として知るべき基礎用語などはある程度似通っている。よって、様々なモデルに共通する前提や基礎的な知識を知ることで、業務集団における共通モデル構築を手助けすることは可能だろう。この文章は、モデルの捉え方や読み方、前提知識などをまとめた「型」や「体系」、いわばWebシステム開発の技術という複雑系モデルに対する、僕自身の「メタモデル」を紹介するものである。

もちろん、メタモデルもモデルの一種なので、前提や観点によって適切に利用できるかどうかは異なる。2023年時点での、Web開発技術を触っているアプリケーションエンジニアの末輩がとらえたモデルなので、ほぼそのまま転用できるところもあれば、そうでないところもある。いろんな人が書いたモデルと比較し、必要な部分を選出・変更することで、自分たちに向いた共有モデル = 「あたりまえ」にカスタマイズすることが最終目的である、ということを心に刻んだうえで読んでもらいたい。

……さて、前置きはこのくらいにして、さっそく本題に入っていこう。
ややこしくも面白い、Web開発技術の「あたりまえ」の世界へようこそ！
